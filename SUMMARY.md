## 1. TL;DR

ShivaGPT is a private dark-mode chat app that talks to **Ollama on the
user's Nvidia DGX Spark ("kailash")**. Phase 1 (file uploads + document/
image Q&A via `qwen2.5vl`) is **code-complete and deployed once**, but a
live debugging session ended with the server becoming unreachable from
the browser at the very end. The latest committed code adds upload
progress + stall detection so a future unreachable state fails fast and
clearly. **Tomorrow's first step is to deploy that and confirm the server
is healthy.** Then proceed to Phase 2 (image generation with Flux.1-dev
via ComfyUI) and Phase 3 (image manipulation with IOPaint).

## 3. Architecture

```
   Safari / Mac dock app                kailash (DGX)
  ┌─────────────────────┐  http :8000  ┌─────────────────────┐  http :11434
  │   index.html (SPA)  │ ──────────►  │ server.py (FastAPI) │ ──────────►  Ollama
  │   localStorage      │ ◄── stream ──│  proxy + extractor  │ ◄── stream ──
  └─────────────────────┘              └─────────────────────┘
```

- Browser only ever talks to one origin (`http://kailash:8000`) — server
  proxies everything to local Ollama on `localhost:11434`. No CORS
  worries on the chat path; CORS is `*` for the dev workflow.
- Streaming uses NDJSON over plain HTTP (works in Safari unlike SSE).
- Chat history, settings, and model-info cache live in `localStorage`.
- Server is run by **systemd** as `shivagpt.service`. Verbose logging is
  currently ON via `Environment=SHIVAGPT_DEBUG=1` in the unit.

## 5. Endpoints

| Method | Path                       | What it does |
|-------:|----------------------------|---|
|    GET | `/`                        | Serves frontend index.html with `Cache-Control: no-store` |
|    GET | `/healthz`                 | `{ok: true, ollama: ...}` if Ollama is reachable |
|    GET | `/api/debug`               | Current debug flag, log level, limits |
|    GET | `/api/models`              | Passthrough Ollama `/api/tags` |
|   POST | `/api/show`                | Passthrough Ollama `/api/show` (used to read each model's `*.context_length`) |
|   POST | `/api/chat`                | Streams Ollama chat back as NDJSON |
|   POST | `/api/files`               | Multipart upload — returns extracted text or base64 image |
|   POST | `/api/image`               | Image manipulation (Upscayl/PIL) — JSON in/out |
|   POST | `/api/imgen`               | Image generation (SDXL Base) — JSON in/out |
|   POST | `/api/codereview`          | Gather code from URL / SSH / local path, stream a review as NDJSON. **Admin auth required.** |
|    GET | `/api/state`               | Admin: read server-side state.json |
|   POST | `/api/state`               | Admin: write server-side state.json |
|   POST | `/api/login`               | Exchange `ADMIN_PASSWORD` for a bearer token |
|    GET | `/manifest.webmanifest`    | PWA manifest |
|    GET | `/icon.svg` `/icon-192.png` `/icon-512.png` | Icons (PNGs generated in-process) |

`/api/files` returns one of:

```json
{ "kind": "text",  "filename": "...", "mime": "...", "text": "...",
  "pages": 12, "pages_extracted": 12, "rows": null, "truncated": false, "size": 412380 }
{ "kind": "image", "filename": "...", "mime": "image/png",
  "base64": "...", "size": 84000 }
```

Caps: 50 MB upload, 200 PDF pages, 200 000 chars text, 60 s server-side processing.

`deepseek-r1:7b`, `qwen2.5vl` matches `qwen2.5vl:32b`, etc. `Settings →
Default model` and per-conversation chips both use this. `visionModel` in
settings (default `qwen2.5vl`) is the auto-switch target.

## 8. Phase 1 — Files + Q&A (DONE)

Built and deployed:

- **Backend**: `POST /api/files` accepts PDF, CSV, TXT/MD/JSON/YAML/XML,
  PNG/JPG/WebP/GIF. PDF uses `pypdf` (run via `asyncio.to_thread` so it
  doesn't block the event loop, with a 60 s `asyncio.wait_for` cap). CSV
  is re-rendered as an aligned text table. Images become base64.
- **Frontend**: paperclip in composer, `<input type=file multiple>` plus
  drag-drop into composer plus paste-image-from-clipboard. Attachment
  chips with thumbnails for images, X to remove, ↻ to retry on error.
  Real upload **progress bar** (XHR-based; fetch can't expose
  `upload.progress` for multipart). Pulsing bar during server-side
  `extracting…` phase.
- **Stall detection**: 3 s pre-flight `/healthz` ping; 12 s no-progress
  watchdog during XHR; 90 s total timeout. Browser shows `Server
  unreachable` immediately if pre-flight fails.
- **Auto-switch to vision model** (`qwen2.5vl`) when an image is
  attached — toast confirms the switch.
- **Inline rendering**: user message in chat shows image thumbnails
  (click to open full-size) and file pills (`PDF · 12 pages`).
- **Send pipeline**: text-file content gets folded into the user message
  as `--- Attached file: NAME ---\n…\n--- end of NAME ---`. Images go
  into Ollama's `messages[].images[]` as raw base64.

## 10b. `/codereview` slash command (LATEST)

**Status: code-complete, structurally verified, not yet deployed by me.**
The user will run `./deploy.sh --service` from their Mac.

Frontend: `/codereview` is in `SLASH_COMMANDS`, handled in
`handleSlashCommandExecution` (parses optional `-model X`, then `<path>`,
then trailing instructions), echoes the command as a user message, and
calls `streamCodeReview()` — a near-copy of `streamChat()` that POSTs
`/api/codereview` with `Authorization: Bearer <state.adminToken>`.

Backend: `POST /api/codereview` in `server.py`, gated by `_check_auth`.
Path dispatch order (matters — `_GIT_SSH_GITHUB_RE` must be checked
before `_SSH_PATH_RE` because both match `git@github.com:owner/repo`):

1. `git@github.com:owner/repo[.git]` → rewritten to GitHub HTTPS, routed
   to `_fetch_github`.
2. `https://github.com/...` → `_fetch_github` (Git Tree API + concurrent
   raw fetches). Honors `GITHUB_TOKEN`.
3. Other `http(s)://` → `_fetch_url` (one text blob).
4. `user@host:/path` → `_fetch_ssh` via blocking `subprocess.run`
   wrapped in `asyncio.to_thread`. Two SSH calls: an enumeration with
   `find ... -prune ... -iname '*.ext' | head -n N`, then a batched
   `cat` with a unique marker so we split the response back into files.
   `BatchMode=yes`, `StrictHostKeyChecking=accept-new`, `ConnectTimeout=8`.
5. Anything else → `_walk_local` on the host running the proxy (DGX).

Env knobs (default in parens):
- `CODEREVIEW_DEFAULT_MODEL` (`deepseek-coder-v2`) — used when no
  `-model` flag.
- `CODEREVIEW_MAX_FILES` (30) — cap on files bundled per review.
- `CODEREVIEW_MAX_CHARS` (120 000) — total character budget.
- `GITHUB_TOKEN` — optional; bumps GitHub rate limits / private repos.

Auth gotcha to remember: the unit doesn't pass `ADMIN_PASSWORD` through,
so on first install it falls back to the baked-in default (`ndr123`).
Override with `sudo systemctl edit shivagpt` →
`Environment=ADMIN_PASSWORD=<long>`. The endpoint can't be reached
without the token, so the gate is real — strength of the gate depends
on this one variable.

SSH gotcha: `ProtectHome=read-only` in the systemd unit means `ssh`
inside the service can *read* the run user's keys but can't *write*
`known_hosts`. Pre-seed once:
`ssh kailash 'ssh-keyscan -H other-box >> ~/.ssh/known_hosts'`. Local
paths and GitHub URLs aren't affected.

The streamed response shape is identical to `/api/chat` (NDJSON,
`{message:{content:"..."}}` per chunk, terminating `{done:true}`), so
the frontend reader in `streamCodeReview` is essentially a copy of
`streamChat` with a different URL/body and the Bearer header.

## 11. Bugs fixed this session (won't bite again)

- iPhone portrait blank → CSS grid `grid-template-columns: 1fr` + safe-area insets.
- iPhone keyboard hides composer → `interactive-widget=resizes-content`
  in viewport meta (replaced JS `position: fixed` workaround).
- iPhone Enter doesn't send → `enterkeyhint="send"` + `beforeinput`
  listener for `insertLineBreak`.
- Composer hidden when messages overflow → `min-height: 0; height: 100%`
  on `.main` and `.sidebar` (CSS grid + flex constraint).
- "Cannot reach server" after deploy → `Cache-Control: no-store` on `/`.
- Upload hangs forever → `asyncio.to_thread` for pypdf + 60 s
  `asyncio.wait_for` server-side; 12 s stall watchdog + pre-flight ping
  client-side.
- `Environment=SHIVAGPT_DEBUG=1` not applied → `install-service.sh` now
  does `systemctl restart` (not just `enable --now`).
- `deploy.sh --service` skipping reinstall → now always re-runs
  `install-service.sh`.

## 12. Diagnostics cheat sheet

```bash
# Is the server running?
curl -i http://kailash:8000/healthz

# Service state + recent logs
ssh -t kailash 'sudo systemctl status shivagpt --no-pager | head -20'
ssh kailash 'sudo journalctl -u shivagpt -n 100 --no-pager'

# Live tail
ssh kailash 'sudo journalctl -u shivagpt -f'

# Verify verbose logging is on
curl -s http://kailash:8000/api/debug | python3 -m json.tool

# Confirm unit file has expected env vars
ssh -t kailash 'sudo grep -i environment /etc/systemd/system/shivagpt.service'

# Full service restart (picks up env var changes from unit)
ssh -t kailash 'sudo systemctl restart shivagpt'

# Reinstall unit + restart (after editing install-service.sh locally)
cd ~/src/shivagpt && ./deploy.sh --service

# Frontend dev loop on Mac (no kailash deploys needed)
cd ~/src/shivagpt && ./dev.sh
# → http://localhost:8000

# To turn debug logging OFF later:
ssh -t kailash 'sudo systemctl edit shivagpt'
# add:
#   [Service]
#   Environment=SHIVAGPT_DEBUG=0
ssh -t kailash 'sudo systemctl restart shivagpt'
```
